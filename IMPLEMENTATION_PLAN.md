# macOS 安卓文件管理器实现计划

## 背景与动机
macOS 原生不支持媒体传输协议 (MTP)，这使得用户仅通过 USB 连接安卓设备时，很难浏览和传输文件。虽然存在一些第三方工具，但构建一个原生的 macOS 应用程序可以提供最佳的性能和用户体验。该工具将支持标准 MTP 连接（针对普通用户）和 ADB 连接（针对启用了 USB 调试的开发者），并封装在现代化的 SwiftUI 界面中。

## 范围与影响
- **平台**: 针对 macOS 13.0+ 的原生 macOS 应用程序。
- **支持的协议**: MTP (媒体传输协议) 和 ADB (安卓调试桥)。
- **核心功能**:
  - 设备检测（USB 插拔）。
  - 文件浏览（列出文件/文件夹，导航目录）。
  - 文件传输（拷贝到 Mac，拷贝到安卓，删除）。
  - 传输进度跟踪。

## 拟定解决方案
### 架构
1.  **前端 (SwiftUI)**: 一个现代化的、类似访达 (Finder) 的界面，包含：
    -   显示已连接安卓设备的侧边栏。
    -   以网格或列表视图显示文件的主要内容区域。
    -   显示正在进行的拷贝操作的状态/传输栏。
2.  **核心领域层**:
    -   `FileProvider` 协议，定义标准操作（`listFiles`, `copyToLocal`, `copyToDevice`, `delete`）。
    -   `DeviceManager` 协议，用于处理连接状态。
3.  **ADB 实现**:
    -   在应用包中捆绑预编译的 macOS 版 `adb` 二进制文件 (`Resources/adb`)。
    -   使用 Swift 的 `Process` (NSTask) 异步执行 CLI 命令。
    -   解析 `adb shell ls -l` 等命令的 stdout 以构建文件模型。
4.  **MTP 实现**:
    -   利用 C 互操作性来桥接类似 `libmtp` 的库或编译为 C 静态库的 Go 后端（如 Kalam）。
    -   创建 C API 的 Swift 封装，以初始化 USB 上下文、声明接口以及读取/写入文件块。

## 已考虑的替代方案
-   **Electron + Node.js**: 更容易进行跨平台开发（如 OpenMTP），但内存占用较高，包体积较大，且非原生 UI 组件。为了获得更好的性能和操作系统集成，已放弃该方案，转而采用原生 Swift。
-   **Rust + Tauri**: 一个很好的轻量级替代方案，但为了获得最佳的 macOS 外观和体验，以及在需要时直接访问 AppKit API，首选原生 SwiftUI。

## 分阶段实施计划

### 第 1 阶段：项目设置与核心抽象
1.  创建一个新的 macOS SwiftUI Xcode 项目。
2.  定义核心领域模型 (`AndroidFile`, `AndroidDevice`)。
3.  定义 `FileProvider` 和 `DeviceManager` 协议。
4.  创建用于 UI 测试的 Mock 实现。

### 第 2 阶段：前端 UI 开发
1.  使用 `NavigationSplitView` 实现主窗口。
2.  构建设备侧边栏组件。
3.  构建文件浏览器组件（列表/网格切换，面包屑导航）。
4.  实现基础的拖放存根。

### 第 3 阶段：ADB 集成
1.  将 macOS 版 `adb` 二进制文件嵌入项目包中。
2.  创建实现 `FileProvider` 协议的 `ADBFileProvider`。
3.  实现 shell 命令执行以及对 `ls`, `pull`, `push` 和 `rm` 的输出解析。
4.  通过 `adb track-devices` 实现设备监控。

### 第 4 阶段：MTP 集成
1.  为 `libmtp`（或等效后端）设置 C 互操作模块映射。
2.  创建实现 `FileProvider` 协议的 `MTPFileProvider`。
3.  通过 C API 实现 USB 设备枚举和连接处理。
4.  使用 MTP 字节流实现文件列出、下载和上传。

### 第 5 阶段：完善与验证
1.  将实际的 Provider 与 SwiftUI 前端集成。
2.  实现并发传输队列和进度报告。
3.  添加错误处理和面向用户的警报（例如“设备已锁定”、“权限被拒绝”）。
4.  确保如果设备在传输过程中断开连接，能够优雅地降级。

## 验证
-   **ADB 测试**: 连接启用 USB 调试的设备。验证是否可以正确列出、拉取和推送文件。
-   **MTP 测试**: 连接处于标准“文件传输”模式（未开启 USB 调试）的设备。验证相同的文件操作是否成功。
-   **UI 测试**: 验证 UI 在大型文件传输期间是否保持响应，并正确反映传输进度。
-   **边缘情况**: 在传输过程中拔掉 USB 线，确保应用能够优雅地处理错误而不会崩溃。

## 迁移与回滚
由于这是一个全新的桌面应用程序，不存在需要迁移的现有系统。如果特定版本发现问题，用户可以回退到之前编译的 `.app` 版本。

## 附录：图片预览核心技术方案

### 1. 方案设计思路
由于 Android 设备的文件系统无法像本地磁盘一样被 macOS 直接挂载（Mount），应用无法通过文件路径直接读取图片数据。本方案采用了**“按需拉取 + 临时缓存 + 异步渲染”**的策略，确保在不占用过多手机资源的情况下提供流畅的预览体验。

### 2. 技术流程
1.  **触发阶段**: 用户双击文件列表中的图片文件，触发 `selectedFile` 状态更新。
2.  **调度阶段**: `FilePreviewPane` 视图挂载并触发 `.onAppear`，启动异步任务调用 `fetchPreviewData`。
3.  **传输阶段**: 
    - 生成本地唯一临时路径（使用 `NSTemporaryDirectory` 和 `UUID`）。
    - 调用后台进程执行 `adb -s <serial> pull <remote_path> <local_path>`。
4.  **渲染阶段**: 
    - 使用 `Data(contentsOf:)` 读取已完成传输的本地文件。
    - 将数据转换为 `NSImage` 对象。
    - 通过 SwiftUI `Image` 容器配合 `.resizable()` 进行自适应渲染。

### 3. 核心技术细节
*   **异步并发模型**: 整个拉取过程封装在 Swift Concurrency (`Task`) 中，避免阻塞 UI 主线程。使用 `MainActor` 确保界面更新的安全。
*   **临时存储管理**: 预览图存储在系统 `T` 目录下，文件名全局唯一，随系统清理机制自动回收，不占用用户长期存储空间。
*   **视图生命周期同步**: 通过 `.id(device.id)` 和 `.onAppear` 钩子，解决了 SwiftUI 条件渲染时状态不同步的边缘情况，确保预览内容与选中文件始终一致。
*   **性能优化**: 采用“懒加载”策略，仅在用户明确交互时才产生 USB 数据传输，极大降低了大规模目录浏览时的功耗和延迟。
